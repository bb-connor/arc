# MERCURY Go-to-Market Strategy

**Date:** 2026-04-04  
**Audience:** Founders, product, sales, and partnership leads

---

## 1. Market Opportunity

MERCURY should be sold as review-grade evidence infrastructure for governed AI
trading workflows.

The opportunity exists because:

- AI already influences trading workflows
- firms lack strong decision-level evidence
- controlled change, rollback, and inquiry workflows need more than scattered
  logs
- existing products do not combine signed records, retained artifacts, and
  portable verification

MERCURY sits between:

- TCA
- surveillance
- LLM observability
- internal workflow logging

That is the category wedge for the first product.

The first commercial wedge is not generic workflow provenance. It is evidence
for one controlled AI change path:

- model, prompt, policy, or parameter release
- rollback authorization
- exception review
- supervisory sign-off
- later inquiry reconstruction

## 4. Sales Motion

### Land motion

The initial sale is a paid pilot that demonstrates:

- signed workflow records
- retained evidence bundles
- proof retrieval and independent verification
- practical value for one change-review, rollback, or inquiry flow

### Expansion motion

After the pilot, close one explicit supervised-live decision record:

- proceed to supervised-live productionization for the same workflow
- defer and gather more evidence in replay or shadow mode
- stop and do not widen scope

That decision should be accompanied by the supervised-live qualification
package so commercial, product, compliance, and workflow owners all review the
same bounded evidence set.

Broader governance, downstream connectors, or embedded distribution should come
only after that bridge closes and only as their own funded decision.

## 5. Positioning and Objections

### Core positioning

MERCURY provides stronger decision provenance than ordinary logs and more
portable evidence than vendor-controlled workflow traces.

The best first description is:

> MERCURY creates signed review-grade evidence for governed AI changes,
> approvals, rollbacks, and inquiries in trading workflows.

### Objection: "This is not a regulatory mandate."

Correct. The value is stronger supervision and investigation evidence, not a
claim that a new law forces immediate purchase.

### Objection: "It does not prove best execution."

Correct. MERCURY records decision provenance and supporting evidence. Best
execution remains a broader supervisory conclusion.

### Objection: "We can build it ourselves."

Some firms can. The practical answer is that publication discipline, verifier
packaging, and retained-evidence integrity are harder to operationalize well
than simple logging.

### Objection: "Why not just use logs?"

Because logs are fragmented, mutable, and hard to present as a coherent,
portable, reviewer-ready workflow record across compliance, risk, client, and
audit contexts.

## 6. Channel and Partner Logic

### Priority 1: design partners

The first objective is referenceable pilot outcomes, not a broad partner
program.

### Priority 2: downstream consumers

Archive, review, surveillance, or governance systems that can consume MERCURY
evidence are useful because they strengthen the buyer story without forcing
deep runtime coupling.

The first downstream case-management lane, the first governance-workbench
lane, the first assurance-suite reviewer family, the first embedded OEM lane,
the first trust-network lane, and the first Chio-Wall lane are now validated.
The current post-Chio-Wall path is one executed multi-product hardening lane over the
existing MERCURY plus Chio-Wall product set:

- one shared Chio service catalog
- one shared Chio dependency map
- one cross-product governance and release matrix for shared versus product-
  local change
- one trust-material recovery drill plus operator alert-routing model
- one fail-closed qualification package over the current product set only

Additional buyer motions, generic barrier-platform breadth, new companion
products, and a merged platform shell remain deferred.

### Priority 3: pull-driven integrations

Supervised-live workflow integration, OMS/EMS, FIX, and market-data
integrations should be built when a funded customer or late-stage prospect
requires the same workflow to move into supervised-live production. Embedded
OEM distribution should come only after the supervised-live bridge is either
operating credibly or explicitly deferred. Trust-network work should come only
after the embedded OEM lane is bounded to one sponsor and one interoperability
surface.

## 7. Success Metrics

### First 12 months

- 2-3 paid pilots
- 1-2 annual conversions
- one clear post-pilot proceed/defer/stop supervised-live decision

### First 24 months

- repeatable pilot-to-platform conversion motion
- at least one credible expansion path inside landed accounts
- buyer references strong enough to support broader distribution

The current bounded path toward those goals is one approved Mercury
reference-distribution lane, one governed Mercury broader-distribution lane,
and one selective-account activation lane, not a generic commercial tooling
program.

## Summary

MERCURY wins by landing as the evidence layer for one governed AI change path,
proving concrete review and inquiry value, converting the same workflow into
supervised-live production only when the proceed/defer/stop record is explicit,
validating one reviewer-facing assurance lane, and then expanding into one
bounded embedded OEM bundle only when buyer pull is specific. The current
post-launch commercial path is now one bounded controlled-adoption lane, one
bounded reference-distribution lane, one bounded broader-distribution lane,
and one bounded selective-account activation lane, not a generic sales or
onboarding platform. Additional partner surfaces, downstream breadth, and
broader OEM paths still require separate buyer pull.
