# MERCURY Phase 4-5 Tickets

**Scope:** Phase 4 (Governance, Downstream Consumers, and Assurance) and Phase 5 (Embedded OEM, Trust Network, and ARC-Wall)  
**Activation:** Begins only after a successful pilot and a funded next step

---

## Phase 4

### E-020: Governance Workbench

#### T-204: Change-review workflow

- Add governed model, prompt, policy, and parameter change-review workflows on
  top of the proof model.

#### T-205: Release, rollback, and exception workflow

- Add governed release, rollback, approval, and exception workflow support.

#### T-206: Governance workflow validation

- Validate the governed workflow experience with workflow owners and control
  teams.

### E-021: Downstream Consumer Integrations

#### T-207: Archive / review connector

- Add one connector into an archive, review, or case-management system.

#### T-208: Surveillance connector

- Add one connector into a surveillance or investigation system.

#### T-209: Downstream distribution validation

- Validate downstream evidence delivery with a partner or design-partner
  target.

### E-022: Assurance Suite

#### T-210: Reviewer package design

- Define reviewer-facing proof and inquiry packages for internal, auditor,
  counterparty, or regulator review.

#### T-211: Reviewer UX and export packaging

- Package the assurance and reviewer experience for the intended population.

#### T-212: Reviewer experience validation

- Test reviewer flows with representative internal and external reviewers.

---

## Phase 5

### E-023: Embedded OEM Distribution

#### T-213: Embedded packaging profile

- Define the contract for embedded or OEM distribution through partner
  platforms.

#### T-214: SDK or partner packaging path

- Define the first true embedded distribution surface beyond downstream
  connectors.

#### T-215: OEM validation

- Validate embedded evidence delivery with a partner or design-partner target.

### E-024: Trust Network

#### T-216: Trust-anchor and witness service design

- Define how trust anchors, witness steps, and publication continuity are
  shared across reviewers or firms.

#### T-217: Shared proof-profile interoperability

- Align MERCURY proof and inquiry profiles with partner and ecosystem
  interoperability needs.

#### T-218: Trust-network rollout plan

- Produce the staged rollout and operating model for a shared trust network.

### E-025: ARC-Wall Core and Buyer Motion

#### T-219: Information-domain evidence schema

- Define the evidence model for tool-boundary control events.

#### T-220: Control-path guard implementation

- Implement the initial information-domain guard or policy evaluation path.

#### T-221: Buyer and packaging validation

- Validate ARC-Wall demand, packaging, and distribution with target buyers.

### E-026: Multi-Product Platform Hardening

**Current activated lane**

- one shared ARC service catalog across MERCURY and ARC-Wall
- one cross-product governance model for release, incident, and trust material
- one bounded platform-hardening backlog for sustained multi-product operation

#### T-222: Shared service boundary review

- Define what remains shared versus product-specific across MERCURY extensions.

#### T-223: Governance and operational model

- Define ownership, release controls, and trust material across products.

#### T-224: Platform hardening backlog

- Publish the infrastructure work required for sustained multi-product support.

---

## Exit Criteria

Phase 4-5 work is complete when:

- governance, downstream, or assurance expansion matches real buyer needs
- embedded OEM, trust-network services, and ARC-Wall reuse the same trust and
  publication model cleanly
- expansion work has a clear operating owner and buyer path
