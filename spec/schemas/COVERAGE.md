## Section E: How to update this file

This file is the orchestrator's audit checklist for milestone M01 Phase 1
schema work. It is regenerated by hand, not by `xtask`, because the
codegen and validation toolchain lands in Phase 3 (not Phase 1). When a
Phase 1 ticket merges to `main`, the merging agent (or the orchestrator)
ticks the corresponding row in section B by replacing `[ ] pending` with
`[x] done` (or `[~] in-progress` while a PR is open). When section A
changes (a new schema lands, a hand-typed file gets a deprecation header,
or a vector domain grows in Phase 2), update the affected table inline and
bump the "Generated" date at the top. Once Phase 3 ships and codegen lands,
this file becomes a candidate for replacement by a machine-generated
manifest under `spec/schemas/MANIFEST.sha256`; until then it is the single
source of truth for "what is in the schema set today" and "what Phase 1 is
still missing".
