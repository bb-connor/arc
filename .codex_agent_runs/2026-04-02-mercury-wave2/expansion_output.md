## 1. Top 5 Insights
- The current macro-order is mostly right, but the post-pilot ordering is not. MERCURY should still land the evidence wedge first, but after pilot the sequence should be `productionized supervised-live use -> governance -> downstream/OEM connectors -> ARC-Wall -> trust network`, not “assurance/governance/OEM” as one block and then “trust network/ARC-Wall.”
- Governance is the strongest first expansion because it is closest to the current wedge, buyer, and proof boundary. The FCA/Bank of England 2024 survey, published December 3, 2025, found 75% of firms using AI, only 34% reporting complete understanding, and 55% of AI use cases involving some automated decision-making while only 2% were fully autonomous; that is exactly the environment where approval, override, exception, and change-review evidence turns into budget.
- OEM is the best ecosystem lever, but not the best first standalone product track. Compliance, archive, surveillance, and case-management vendors are already building AI-native investigation workflows and open APIs, so MERCURY can become their evidence substrate; that is strong once one internal workflow is already working credibly, but dangerous if pursued before direct customer pull.
- Assurance matters, but it should be treated as a horizontal packaging/export capability first, not a standalone early expansion line. ISO/IEC 42001, IOSCO’s March 12, 2025 AI-in-capital-markets work, the EU AI Act rollout, and IAASB’s February 10, 2026 feedback all point to rising auditability and governance demand, but the immediate buyer is still usually the internal governance owner rather than an external assurance market buying separate software.
- ARC-Wall is more commercially promising than Trust Network in the medium term. Microsoft’s March 20, 2026 and January 29, 2026 updates show enterprise spending is moving quickly toward agent controls, DLP, audit, and tool-boundary oversight; by contrast, trust-network infrastructure is improving, but still looks like an ecosystem-enablement bet. Inference: W3C VC 2.0 and GLEIF vLEI show standards maturity improving, but not yet enough buyer pull to make Trust Network a first post-pilot motion.

## 2. Top 3 Risks
- MERCURY could drift into generic “AI governance” or “AI assurance” software and lose its sharp proof-boundary advantage against larger GRC, model-risk, and compliance incumbents.
- OEM too early could turn the company into a bespoke integration shop or a feature inside someone else’s platform, especially given third-party dependence is already material in financial-services AI deployments.
- Pursuing Trust Network or a standalone ARC-Wall motion before one repeatable post-pilot expansion is working would create a multi-sided go-to-market problem too early.

## 3. Top 3 Expansion Paths in Order
1. **Governance Workbench**, but anchored by a new explicit supervised-live productionization step. This is the cleanest path from pilot evidence to recurring budget because it keeps the same buyer, same workflow, and same proof boundary while moving from shadow/replay to real review operations.
2. **OEM / downstream evidence distribution** into surveillance, archive, review, and case-management systems. This is the strongest ecosystem move because it lets MERCURY live inside existing systems of work and can widen distribution without forcing MERCURY to build a full reviewer UI stack.
3. **ARC-Wall**. It is a better third bet than Assurance or Trust Network because tool-boundary and information-domain control for agents is becoming an urgent problem, and it reuses the same ARC signing, publication, and verification substrate.

## 4. Roadmap Changes
- Insert a `Phase 3.5: Productionization / Supervised Live Workflow` between pilot readiness and current Phase 4.
- Reorder current Phase 4 into `4A Governance`, `4B Downstream Connectors/OEM`, and `4C Assurance Exports`.
- Reframe Assurance as a capability layer on top of governance and OEM: proof packaging, redaction, reviewer bundles, communications controls, and auditor/client exports.
- Decouple ARC-Wall from Trust Network and move ARC-Wall ahead of Trust Network when there is a named banking, barrier, or security sponsor.
- Push Trust Network to a later phase with explicit gates: at least two external verifier ecosystems, or one credible ecosystem anchor partner, before building shared trust-anchor services.
