interface types {

}

const BLOCKED_TOOLS: ReadonlySet<string> = new Set(["dangerous_tool", "rm_rf", "drop_database"]);
export function evaluate(request: GuardRequest): Verdict {
  if (BLOCKED_TOOLS.has(request.toolName)) {
    return { tag: "deny", val: "tool is blocked by policy" };
  }
  return { tag: "allow" };
}

Create build scripts and compile the example guard through the full pipeline:

1. **scripts/generate-types.sh** (chmod +x): Generate Go bindings from WIT.
   The pipeline is:
   a. Use `wkg wit build` to resolve WIT dependencies (WASI types) and produce
      a bundled WIT package:
      ```bash
      wkg wit build -d wit/ -o dist/arc-guard-go.wasm
      ```
      If `wkg` is not available, manually fetch WASI WIT files or use
      `wasm-tools compose` as an alternative.
   b. Generate Go bindings:
      ```bash
      go tool wit-bindgen-go generate \
          --world guard \
          --out internal \
          dist/arc-guard-go.wasm
      ```
      If `go tool wit-bindgen-go` is not available, install it via
      `go get -tool go.bytecodealliance.org/cmd/wit-bindgen-go`.
   c. Set `--package-root` if needed to match the go.mod module path.

2. **scripts/build-guard.sh** (chmod +x): Full build pipeline:
   - Step 1: Call generate-types.sh (or inline the commands)
   - Step 2: Create `dist/` directory if missing
   - Step 3: Compile with TinyGo:
     ```bash
     tinygo build \
         -target=wasip2 \
         -no-debug \
         --wit-package dist/arc-guard-go.wasm \
         --wit-world guard \
         -o dist/tool-gate-raw.wasm \
         ./examples/tool-gate/
     ```
   - Step 4: Strip WASI imports with wasi-virt:
     ```bash
     wasi-virt dist/tool-gate-raw.wasm -o dist/tool-gate.wasm
     ```
   - Step 5: Verify the output has zero imports:
     ```bash
     wasm-tools component wit dist/tool-gate.wasm
     ```
   - Step 6: Print output file size

   CRITICAL: The wasi-virt step is mandatory. TinyGo's wasip2 output will
   import `wasi:cli` interfaces. The ARC guard host has an empty linker and
   will reject components with WASI imports. wasi-virt satisfies these imports
   with no-op virtualizations, producing a component with zero imports.

3. **Run the build**: Execute the build pipeline. This involves multiple
   external tools (wkg, wit-bindgen-go, tinygo, wasi-virt, wasm-tools).
   Install any missing tools:
   - `brew install tinygo` (or follow https://tinygo.org/getting-started/)
   - `cargo install --git https://github.com/bytecodealliance/wasi-virt`
   - `cargo install --locked wasm-tools@1.225.0`
   - `go get -tool go.bytecodealliance.org/cmd/wit-bindgen-go`

   If any tool is unavailable and cannot be installed in the environment,
   document the exact commands needed and mark the build as requiring those
   tools. The scripts should still be correct and executable when the tools
   are present.

4. **Fix import paths**: After wit-bindgen-go generates bindings into
   `internal/`, update `examples/tool-gate/main.go` import paths to match
   the actual generated package structure. The generated paths depend on the
   WIT package name and module path.

5. **Verify zero imports**: After wasi-virt, the component should have zero
   imports. If wasi-virt does not fully strip all WASI imports, document which
   imports remain and investigate alternative wasi-virt flags.

   Expected binary size: 500 KiB - 2 MiB (TinyGo runtime is lightweight).
