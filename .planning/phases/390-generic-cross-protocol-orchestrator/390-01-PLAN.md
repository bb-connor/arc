# Plan 390-01

1. Introduce a dedicated shared cross-protocol runtime home under `crates/`
   for `DiscoveryProtocol`, truthful `BridgeFidelity`,
   `CrossProtocolCapabilityRef`, `CrossProtocolCapabilityEnvelope`,
   `CrossProtocolTraceContext`, hop metadata, and `CapabilityBridge` traits so
   A2A and ACP stop redefining bridge contracts independently.
2. Implement `CrossProtocolOrchestrator` on top of the existing `ArcKernel`
   tool-call and receipt surfaces so it can accept a bridged execution
   request, extract or inject capability context, enforce attenuation
   fail-closed, dispatch through the kernel, and attach shared bridge-lineage
   receipt metadata.
3. Route one authoritative A2A execution path and one authoritative ACP
   execution path through the orchestrator to prove the shared substrate in
   live code, while leaving final compatibility-path quarantine and default-path
   cleanup to phase `391`.
4. Add focused tests for capability reference propagation, attenuation subset
   enforcement, truthful receipt-lineage metadata, and one orchestrated
   A2A-to-kernel plus ACP-to-kernel flow.

## Status

Completed on 2026-04-14. This plan landed the shared `arc-cross-protocol`
substrate, moved the default authoritative A2A and ACP execution paths onto
`CrossProtocolOrchestrator`, and added focused lineage/attenuation tests.
