2. Implement `cmd_guard_test(wasm_path: &Path, fixture_paths: &[PathBuf], fuel_limit: u64) -> Result<(), CliError>`:
   - Read and verify the .wasm bytes exist
   - For each fixture path: read the YAML file, parse as `Vec<TestFixture>`
   - Create a `WasmtimeBackend`, call `load_module` with the wasm bytes and fuel_limit
   - For each fixture:
     - Call `backend.evaluate(&fixture.request)`
     - Compare verdict to `fixture.expected_verdict` ("allow" matches `GuardVerdict::Allow`, "deny" matches `GuardVerdict::Deny { .. }`)
     - If deny and `deny_reason_contains` is set, check the deny reason contains the expected substring
     - Track pass/fail count, print per-fixture results: `[PASS] fixture_name` or `[FAIL] fixture_name: expected X, got Y`
   - After all fixtures: print summary `N passed, M failed out of T total`
   - If any fixture failed, return `Err(CliError::Other("N test(s) failed".into()))`
   - IMPORTANT: After each fixture evaluation, create a fresh WasmtimeBackend and reload the module. The wasmtime backend's Store carries per-evaluation state (fuel counter, linear memory). Reusing the backend across fixtures means fuel from previous runs is not properly reset. Create one backend per fixture OR find a way to reset state. The simplest correct approach: create a new backend + load_module for each fixture.

   - NOTE: Integration tests that actually load .wasm require the wasmtime feature and a compiled .wasm binary. The unit tests focus on fixture parsing. The verify step uses `cargo test -p arc-cli` to confirm parsing tests pass. Full integration testing happens via manual `arc guard test` invocation.
