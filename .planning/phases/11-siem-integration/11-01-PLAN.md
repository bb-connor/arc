  <files>crates/pact-siem/src/manager.rs</files>

  <action>
    1. Define `SiemConfig`:
       - `db_path: PathBuf` -- path to the receipt SQLite database
       - `poll_interval: Duration` -- how often to poll for new receipts (default: 5 seconds)
       - `batch_size: usize` -- max receipts per poll (default: 100)
       - `max_retries: u32` -- retry attempts per exporter before DLQ (default: 3)
       - `base_backoff_ms: u64` -- base backoff for exponential retry (default: 500)
       - `dlq_capacity: usize` -- max DLQ entries (default: 1000)

    2. Define `SiemError` enum with variants: DbError(String), ConfigError(String)

    3. Implement `ExporterManager`:
       - Fields: `exporters: Vec<Box<dyn Exporter>>`, `dlq: DeadLetterQueue`, `cursor: u64`, `config: SiemConfig`
       - `ExporterManager::new(config: SiemConfig) -> Self` -- creates DLQ from config.dlq_capacity, sets cursor to 0, empty exporters vec
       - `add_exporter(&mut self, exporter: Box<dyn Exporter>)` -- pushes exporter to vec
       - `pub async fn run(&mut self, cancel: tokio::sync::watch::Receiver<bool>)` -- main loop:
         a. Create tokio::time::interval(config.poll_interval)
         b. Each tick: open rusqlite::Connection read-only to config.db_path (wrap in spawn_blocking)
         c. Execute duplicated SQL: `SELECT seq, raw_json FROM pact_tool_receipts WHERE seq > ?1 ORDER BY seq ASC LIMIT ?2` with cursor and batch_size
         d. Build Vec<SiemEvent> via SiemEvent::from_receipt for each row
         e. For each exporter: call export_batch with exponential backoff retry (base_backoff_ms * 2^attempt, up to max_retries)
         f. On all retries exhausted: serialize failed events to JSON, push FailedEvent to DLQ, log tracing::warn with exporter name and error
         g. CRITICAL: advance cursor past the batch after all exporters have processed (whether successful or DLQ'd) -- do not re-fetch the same events. Log tracing::info with seq range if any events were DLQ'd.
         h. Check cancel receiver -- break loop if true

    4. The cursor should NOT be persisted to disk in this implementation. It starts at 0 on each process start, meaning restart re-exports all receipts. This is acceptable because both Splunk HEC (uses timestamp dedup) and Elasticsearch (uses _id upsert) handle duplicates idempotently. A future phase can add cursor persistence.

    CRITICAL: Use spawn_blocking for all rusqlite calls. The Connection is opened per-poll rather than kept open, to avoid holding a read lock across polls. If performance is a concern in the future, this can be changed to a persistent read connection (WAL mode readers do not block writers).
  </action>
