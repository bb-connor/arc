1. **Modify `check_and_increment_budget`** to handle monetary budgets. The method currently calls `try_increment`. When a matching grant has monetary fields (`max_cost_per_invocation` or `max_total_cost`), call `try_charge_cost` instead. The updated logic:

```rust
fn check_and_increment_budget(
    &mut self,
    cap: &CapabilityToken,
    matching_grants: &[MatchingGrant<'_>],
) -> Result<(usize, Option<BudgetChargeResult>), KernelError> {
    // Returns (matched_grant_index, Option<monetary charge result>)
    // BudgetChargeResult carries the info needed for FinancialReceiptMetadata
```

Create a `BudgetChargeResult` struct (private to lib.rs):
```rust
struct BudgetChargeResult {
    grant_index: usize,
    cost_charged: u64,
    currency: String,
    budget_remaining: u64,
    budget_total: u64,
}
```

For each matching grant:
- If grant has `max_cost_per_invocation` or `max_total_cost`, call `try_charge_cost` with the grant's monetary limits. Use `max_cost_per_invocation.as_ref().map(|m| m.units)` as the `cost_units` parameter (worst-case debit pre-execution).
- If grant has only `max_invocations` (no monetary), call `try_increment` as before.
- Return the matched grant index so it can be passed to GuardContext.

2. **Populate `matched_grant_index` in `run_guards`**. Update the `run_guards` call sites in both `evaluate_tool_call_with_session_roots` and `evaluate_tool_call_with_nested_flow_client`:
```rust
// After check_and_increment_budget returns the matched index:
let (matched_index, charge_result) = self.check_and_increment_budget(cap, &matching_grants)?;

// Pass to run_guards via GuardContext:
let ctx = GuardContext {
    request,
    scope: &cap.scope,
    agent_id: &request.agent_id,
    server_id: &request.server_id,
    session_filesystem_roots,
    matched_grant_index: Some(matched_index),
};
```

3. **Use `invoke_with_cost` in dispatch_tool_call** when the matched grant has monetary fields. Create a helper:
```rust
fn dispatch_tool_call_with_cost(
    &self,
    request: &ToolCallRequest,
    has_monetary_grant: bool,
) -> Result<(ToolServerOutput, Option<ToolInvocationCost>), KernelError> {
    let server = self.tool_servers.get(&request.server_id)...;
    if has_monetary_grant {
        let (value, cost) = server.invoke_with_cost(&request.tool_name, request.arguments.clone(), None)?;
        Ok((ToolServerOutput::Value(value), cost))
    } else {
        // existing dispatch path
    }
}
```

4. **After successful invocation, adjust the budget charge if actual cost differs from worst-case**. If `invoke_with_cost` returns `Some(cost)` and `cost.units < max_cost_per_invocation`, the kernel already charged the max -- this is fine for Phase 8 (no refund mechanism). If `cost.units > max_cost_per_invocation`, log a warning and set `settlement_status = "failed"`.

5. **Populate FinancialReceiptMetadata on allow responses** with monetary grants. In `build_allow_response` or `finalize_tool_output`, when the grant has monetary fields:
```rust
let financial_meta = FinancialReceiptMetadata {
    grant_index: matched_index as u32,
    cost_charged: actual_cost_or_max,
    currency: grant_currency.clone(),
    budget_remaining: budget_total - total_charged,
    budget_total,
    delegation_depth: cap.delegation_chain.len() as u32,
    root_budget_holder: cap.issuer.to_hex(),
    payment_reference: None,
    settlement_status: if overrun { "failed" } else { "authorized" }.to_string(),
    cost_breakdown: reported_cost.and_then(|c| c.breakdown),
    attempted_cost: None,
};
let metadata = Some(serde_json::json!({ "financial": financial_meta }));
```

7. **Add checkpoint triggering to `record_arc_receipt`**. Add state to ArcKernel:
```rust
checkpoint_batch_size: u64,       // default 100
checkpoint_seq_counter: u64,      // monotonic checkpoint counter
last_checkpoint_seq: u64,         // seq of last receipt in previous batch
```

In `record_arc_receipt`, when receipt_store is a SqliteReceiptStore, call `append_arc_receipt_returning_seq`. When `(seq - last_checkpoint_seq) >= checkpoint_batch_size`, trigger a checkpoint:
- Fetch `receipts_canonical_bytes_range(last_checkpoint_seq + 1, seq)`
- Call `build_checkpoint(next_checkpoint_seq, batch_start, batch_end, bytes, keypair)`
- Call `store_checkpoint(checkpoint)`
- Update `last_checkpoint_seq = seq` and increment `checkpoint_seq_counter`

For the receipt_store type, use downcasting or add a trait method. The simplest approach: add `checkpoint_batch_size` to `KernelConfig` and track checkpoint state on ArcKernel. Use the existing `tool_receipt_count()` or the returned seq to detect boundaries.

8. **Add integration tests** covering all behavior items. Create test helper `MonetaryCostServer` that implements `ToolServerConnection` with `invoke_with_cost` returning specific costs. Test the full pipeline from capability issuance through denial.

IMPORTANT: Do not break existing tests. The changes to `check_and_increment_budget` return type and `GuardContext` must be backward compatible. Existing non-monetary grants should continue working exactly as before.
