  <action>
    Introduce a new shared identity representation without breaking the existing bearer alpha surface:

    1. In `crates/pact-core/src/session.rs`, add:
       - `EnterpriseFederationMethod` enum with `Jwt`, `Introspection`, `Scim`, `Saml`
       - `EnterpriseIdentityContext` struct with fields:
         - `provider_id: String`
         - `provider_record_id: Option<String>`
         - `provider_kind: String`
         - `federation_method: EnterpriseFederationMethod`
         - `principal: String`
         - `subject_key: String`
         - `client_id: Option<String>`
         - `object_id: Option<String>`
         - `tenant_id: Option<String>`
         - `organization_id: Option<String>`
         - `groups: Vec<String>`
         - `roles: Vec<String>`
         - `source_subject: Option<String>`
         - `attribute_sources: BTreeMap<String, String>`
         - `trust_material_ref: Option<String>`
       - add optional `enterprise_identity` to the OAuth bearer `SessionAuthMethod` payload with serialized name `enterpriseIdentity`

    2. Keep `OAuthBearerFederatedClaims` intact for backward compatibility. Do not remove or rename `federatedClaims`.

    3. In `crates/pact-cli/src/remote_mcp.rs`:
       - add a helper that converts the existing normalized bearer principal plus `OAuthBearerFederatedClaims` into `EnterpriseIdentityContext`
       - add a concrete helper `derive_enterprise_subject_key(provider_scope: &str, canonical_principal: &str) -> Result<String, CliError>` and use it for bearer federation
       - set `provider_id` to the configured provider record id when available, otherwise fall back to the issuer string
       - set `provider_record_id` only when a validated provider-admin record matched the request
       - set `provider_kind` to exactly `oidc_jwks` or `oauth_introspection` based on the configured source
       - populate `attribute_sources` with the source attribute names used for principal, tenant, organization, groups, and roles
       - populate the new `enterpriseIdentity` field whenever bearer federation is active
       - define and test the exact enterprise-lane boundary: bearer federation without a validated provider record still emits `enterpriseIdentity` for observability but remains on the legacy bearer admission path

    4. Extend `crates/pact-cli/tests/mcp_serve_http.rs` so the existing bearer/admin trust tests assert:
       - `enterpriseIdentity.principal` matches the current bearer-derived principal
       - `enterpriseIdentity.subjectKey` is stable for the same provider scope plus canonical principal
       - `enterpriseIdentity.tenantId`, `organizationId`, `groups`, and `roles` mirror existing `federatedClaims`
       - `enterpriseIdentity.attributeSources` identifies the source attributes used to normalize principal/group/role values
       - bearer requests without a validated provider record keep current legacy behavior while still surfacing `enterpriseIdentity`
       - existing `federatedClaims` assertions still pass unchanged
  </action>

  <action>
    Add fail-closed normalization helpers for non-bearer enterprise identity inputs:

    1. Define structured input types in `enterprise_federation.rs`:
       - `ScimIdentityInput` with `user_id`, `external_id`, `user_name`, `tenant_id`, `organization_id`, `groups`, `roles`, and generic `attributes: BTreeMap<String, String>`
       - `SamlIdentityInput` with `name_id`, `session_index`, `tenant_id`, `organization_id`, `groups`, `roles`, and generic `attributes: BTreeMap<String, String>`

    2. Implement:
       - `normalize_scim_identity(provider: &EnterpriseProviderRecord, input: &ScimIdentityInput) -> Result<EnterpriseIdentityContext, CliError>`
       - `normalize_saml_identity(provider: &EnterpriseProviderRecord, input: &SamlIdentityInput) -> Result<EnterpriseIdentityContext, CliError>`
       - both helpers must call the shared `derive_enterprise_subject_key` function so subject keys stay stable across bearer, SCIM, and SAML for the same provider scope plus canonical principal

    3. Use `provider.subject_mapping.principal_source` exactly as the canonical principal selector:
       - if it is `user_id`, use `user_id`
       - if it is `external_id`, use `external_id`
       - if it is `user_name`, use `user_name`
       - if it is `name_id`, use `name_id`
       - otherwise look the field name up in `attributes`
       - if the resolved value is missing or blank, return `CliError` and do not emit any partial identity context

    4. Normalize groups and roles with the same trim/sort/dedup behavior used for bearer claim lists.

    5. Set `provider_kind` exactly to `scim` or `saml`, and `federation_method` exactly to `Scim` or `Saml`.
       - populate `attribute_sources` for every resolved principal, tenant, organization, group, and role field
       - carry `trust_material_ref` from the validated provider record into the emitted identity context

    6. Add unit tests for:
       - same provider id + same canonical principal => same derived subject key across bearer, SCIM, and SAML normalization
       - different canonical principal => different derived subject key
       - missing principal_source field => error
       - blank/whitespace-only group and role values are removed
       - ambiguous SAML input with missing `name_id` when `principal_source = name_id` returns an error
       - attribute source provenance is preserved on the emitted enterprise identity context

    7. Do not add an XML parser in this plan unless it is required to satisfy the normalization tests. If SAML tests can operate on structured input fixtures, keep the dependency surface unchanged.
  </action>
