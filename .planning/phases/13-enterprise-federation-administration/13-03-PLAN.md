    Extend the existing origin policy model instead of creating a second enterprise-specific evaluator:

    1. In `crates/arc-policy/src/models.rs` and `crates/arc-policy/src/evaluate.rs`, add first-class origin fields:
       - `organization_id: Option<String>`
       - `groups: Vec<String>`
       - `roles: Vec<String>`

    2. Keep `actor_role` for backward compatibility, but do not use it as a fallback replacement for the new `roles` field. If a rule specifies `roles`, matching must use the new multi-value field only.

    3. Update `match_origin` in `evaluate.rs` with these exact semantics:
       - if `organization_id` is set on the rule and origin does not match exactly, return `None`
       - if `groups` is non-empty on the rule, every listed group must be present in `origin.groups`; otherwise return `None`
       - if `roles` is non-empty on the rule, every listed role must be present in `origin.roles`; otherwise return `None`
       - scoring should increase for organization, groups, and roles so more specific enterprise rules outrank provider-only rules

    4. Add unit tests covering:
       - provider + tenant + organization exact match
       - missing organization denies
       - required group subset match succeeds
       - missing required group denies
       - required role subset match succeeds
       - missing required role denies
       - actor_role-only legacy rule still works when `roles` is absent

    5. Re-export any new types through `crates/arc-policy/src/lib.rs` so trust-control code can build the richer `OriginContext`.
  </action>

    1. In `crates/arc-cli/src/trust_control.rs`, define `EnterpriseAdmissionAudit` with these fields:
       - `provider_id: String`
       - `provider_record_id: Option<String>`
       - `provider_kind: String`
       - `federation_method: String`
       - `principal: String`
       - `subject_key: String`
       - `subject_public_key: String`
       - `tenant_id: Option<String>`
       - `organization_id: Option<String>`
       - `groups: Vec<String>`
       - `roles: Vec<String>`
       - `attribute_sources: BTreeMap<String, String>`
       - `trust_material_ref: Option<String>`
       - `matched_origin_profile: Option<String>`
       - `decision_reason: Option<String>`

    2. Add a helper that converts `EnterpriseIdentityContext` into `arc_policy::OriginContext` using exact field mapping:
       - `provider` = `provider_id`
       - `tenant_id` = `tenant_id`
       - `organization_id` = `organization_id`
       - `groups` = `groups`
       - `roles` = `roles`

    3. In the federated issue admission path:
       - define the enterprise-provider lane precisely: it is active only when `enterpriseIdentity.provider_record_id` resolves to a validated provider-admin record or when SCIM/SAML admission is invoked with a validated provider record
       - if enterprise-provider lane is not active, preserve the existing bearer-only admission path and add regression coverage that proves no behavior change
       - if a validated enterprise provider and `enterpriseIdentity` context are present, evaluate admission using the richer `OriginContext`
       - if enterprise-provider lane is active but provider validation fails or required fields are missing, deny instead of falling back to a weaker legacy path
       - capture the matched origin profile and decision reason in `EnterpriseAdmissionAudit`

    4. Extend `FederatedIssueResponse` with optional `enterprise_audit: Option<EnterpriseAdmissionAudit>`.

    5. In `crates/arc-cli/tests/federated_issue.rs`, add/extend tests proving:
       - allow when provider, tenant, organization, groups, and roles satisfy policy
       - deny when organization mismatches
       - deny when a required group is missing
       - deny when a required role is missing
       - bearer enterprise observability without a validated provider record still follows the legacy bearer-only path
       - enterprise-provider lane active + invalid provider record does not silently fall back to weaker admission
       - allow response carries `enterprise_audit` with provider id, provider record id, principal, subject key, provenance fields, and matched origin profile
