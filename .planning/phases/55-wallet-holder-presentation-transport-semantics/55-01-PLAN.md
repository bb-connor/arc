# Plan 55-01

Define one conservative holder-facing presentation transport contract over the
existing ARC challenge and response artifacts.

## Scope

- add typed transport metadata that can advertise holder-safe challenge fetch
  and response submission URLs
- define how the transport binds to stored challenge ids and verifier-owned
  replay state
- keep the contract ARC-specific and compatible with the current challenge and
  response schemas

## Success Criteria

- ARC has one explicit typed holder transport contract for challenge fetch and
  response submission
- the transport is layered over the existing challenge and response truth
  rather than inventing a second presentation artifact
- validation fails closed on missing challenge ids or malformed transport
  metadata
