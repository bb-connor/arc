# Requirements: ARC v2.3 Production and Standards

**Defined:** 2026-03-25
**Core Value:** ARC must provide deterministic, least-privilege agent access
with auditable outcomes, and produce cryptographic proof artifacts that enable
economic metering, regulatory compliance, and portable trust across
organizational boundaries.

## v2.3 Requirements

### Release Hygiene and Structure

- [x] **PROD-07**: Release inputs exclude generated/build/cache artifacts, and
  package/release outputs are reproducible from source rather than committed
  build products.
- [x] **PROD-08**: Oversized runtime and CLI surfaces are split into
  maintainable module boundaries with clearer ownership and no behavioral
  regressions.

### Qualification and Operations

- [x] **PROD-09**: A documented release-qualification lane passes across the
  Rust workspace, dashboard, SDK packaging, and conformance-critical checks.
- [x] **PROD-10**: Deployment, configuration, backup/restore, and
  upgrade/rollback runbooks exist for trust-control and the primary operator
  surfaces.

### Observability and Specification

- [x] **PROD-11**: Production observability and health surfaces are sufficient
  to diagnose trust-control, remote runtime, A2A, and federation failures
  without ad hoc code inspection.
- [x] **PROD-12**: `spec/PROTOCOL.md` v2 accurately describes shipped receipts,
  portable trust, federation, A2A, and certification behavior, including
  compatibility expectations.

### Standards and Launch Readiness

- [x] **PROD-13**: SDKs, examples, and operator docs align with the v2 spec and
  supported production workflows.
- [x] **PROD-14**: Standards-submission artifacts and a GA launch checklist
  exist for receipts and portable trust, including known gaps, terminology, and
  non-goals.

## Future Requirements

### v2.4 Commercial Trust Primitives

- **COMM-01**: Insurer-facing behavioral feed is defined and exportable.
- **COMM-02**: Marketplace trust primitives build on ARC capabilities,
  receipts, and portable reputation.
- **COMM-03**: Reputation federation works across org boundaries without
  inventing unsupported trust claims.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full public certification marketplace | Not required for productionizing the current product surface. |
| Payment-rail settlement integration | Separate economics/commercial milestone. |
| Multi-region Byzantine consensus | Premature for the current deployment and trust model. |
| Full OS sandbox manager | Different product layer from protocol and trust-control productionization. |
| Net-new major trust product surfaces | v2.3 is a hardening and launch-readiness milestone, not a breadth milestone. |
