# Requirements: PACT v2.1 Federation and Verifier Completion

**Defined:** 2026-03-24
**Core Value:** PACT must provide deterministic, least-privilege agent access with auditable outcomes, and produce cryptographic proof artifacts that enable economic metering, regulatory compliance, and portable trust across organizational boundaries.

## v1 Requirements

### Federation Identity

- [x] **FED-01**: Operator can onboard enterprise identity sources through provider-admin surfaces that cover SCIM and SAML-backed federation inputs rather than only raw bearer verifier settings.
- [x] **FED-02**: Trust-control policy can allow or deny portable-trust admission using provider, tenant, organization, role, and group context without silently widening trust.
- [x] **FED-03**: Cross-org delegated issuance preserves upstream identity provenance through imported evidence, local delegation anchors, and downstream operator analytics.

### Verifier Infrastructure

- [x] **VER-01**: Verifier policy can be created, signed, distributed, and referenced as a reusable artifact instead of only embedded inline in a challenge.
- [x] **VER-02**: Verifier challenge replay state is persisted and enforced so consumed challenges cannot be replayed across processes or restarts.
- [x] **VER-03**: Challenge create/respond/verify flows work consistently across local CLI and remote verifier surfaces with explicit transport and state semantics.

### Passport Composition

- [x] **PASS-01**: Multi-issuer passport composition semantics are explicitly defined and enforced without weakening current single-credential verification guarantees.
- [x] **PASS-02**: Verifier evaluation reports acceptance and rejection at the issuer and credential level for composed portable-trust bundles.

### Cross-Org Analytics

- [x] **XORG-01**: Operators can reference shared remote evidence packages in trust-control and dashboard without importing the full package into local receipt storage.
- [x] **XORG-02**: Cross-org operator reports and comparison/drift views attribute activity across local and remote delegation chains truthfully.

## v2 Requirements

### Ecosystem Hardening

- **A2A-01**: Remaining A2A auth matrix is covered, including provider-specific and non-header schemes not yet shipped.
- **A2A-02**: A2A task lifecycle coverage extends beyond current send/get/subscribe/cancel plus push-config CRUD to deeper long-running operational semantics.
- **CERT-01**: Certification registry/storage exists for signed certification artifacts and external tool-server entries.

### Production and Standards

- **PROD-07**: Protocol specification v2 matches the shipped portable-trust and federation surface.
- **PROD-08**: Production deployment, runbook, and scale-hardening surfaces are complete enough for broader launch.
- **PROD-09**: Standards-submission artifacts exist for the receipt and portable-trust model.

### Commercial Trust Primitives

- **COMM-01**: Insurer-facing behavioral feed is defined and exportable.
- **COMM-02**: Marketplace trust primitives build on PACT capabilities, receipts, and portable reputation.
- **COMM-03**: Reputation federation works across org boundaries without inventing unsupported trust claims.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Custom payment rail | PACT is authorization and attestation, not settlement infrastructure. |
| Full OS sandbox manager | Root and capability enforcement remain the product boundary. |
| Multi-region Byzantine consensus | Premature for the current deployment and trust model. |
| Protocol-level anti-collusion prevention | Detection and evidence surfaces come first; prevention remains research. |
| Insurer and marketplace commercialization in v2.1 | This milestone closes portable-trust infrastructure, not downstream business surfaces. |
