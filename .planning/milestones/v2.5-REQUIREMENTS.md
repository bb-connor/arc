# Requirements: Active v2.5 and Planned v2.6+

**Defined:** 2026-03-25  
**Latest archived milestone:** v2.4 Architecture and Runtime Decomposition
(archived 2026-03-25)  
**Active milestone:** v2.5 ARC Rename and Identity Realignment  
**Next planned milestone:** v2.6 Governed Transactions and Payment Rails  
**Core Value:** ARC must provide deterministic, least-privilege agent
authority with auditable outcomes, bounded spend, and cryptographic proof
artifacts that enable economic security, regulatory compliance, and portable
trust across organizational boundaries.

## Active Milestone: v2.5 ARC Rename and Identity Realignment

### Product Identity and Surface Rename

- [x] **ARC-01**: Repository metadata, top-level docs, release artifacts,
  dashboards, examples, and operator-facing product language use `ARC` as the
  primary name instead of `ARC`.
- [x] **ARC-02**: CLI and runtime operator surfaces migrate from `arc` to
  `arc`, with explicit compatibility aliases or a documented major-version
  break strategy instead of silent drift.
- [x] **ARC-03**: Rust crate package names, SDK package names, and public
  import surfaces move from `arc-*` / `@arc-protocol/*` to ARC-aligned names
  with a downstream migration map.

### Protocol, Artifact, and Identity Compatibility

- [x] **ARC-04**: Protocol/profile/schema identifiers, headers, route prefixes,
  file markers, env vars, and config keys are fully inventoried and each is
  either renamed to ARC or explicitly frozen with rationale.
- [x] **ARC-05**: Legacy ARC receipts, passports, verifier policies,
  certifications, evidence packages, and policy inputs remain verifiable,
  importable, or convertible under a documented compatibility contract.
- [x] **ARC-06**: The `did:arc` transition strategy is resolved into one
  coherent contract (`did:arc`, dual-resolution, frozen legacy method, or
  equivalent) and docs/code align to that decision.

### Delivery, Documentation, and Qualification

- [x] **ARC-07**: README, VISION, Strategic Roadmap, standards drafts, release
  docs, and operator guides all tell one ARC-centered story aligned to the
  economic-security framing from `docs/research/DEEP_RESEARCH_1.md`.
- [x] **ARC-08**: Qualification, conformance, SDK packaging, and migration docs
  prove the renamed ARC surface and any supported legacy PACT aliases
  end-to-end.

## Next Planned Milestone: v2.6 Governed Transactions and Payment Rails

- [ ] **ECON-01**: Governed transaction intent and approval artifacts exist as
  first-class policy inputs and receipt fields.
- [ ] **ECON-02**: x402 pay-per-API integration works with truthful budget,
  payment, and receipt semantics.
- [ ] **ECON-03**: ACP / Shared Payment Token bridge works with bounded-spend
  semantics and linked execution receipts.
- [ ] **ECON-04**: Pending/failed settlement reconciliation is operator-visible
  without falsifying tool execution outcomes.
- [ ] **ECON-05**: ARC supports at least one richer budget profile beyond money
  alone without collapsing the semantics.

## Future Planned Milestone: v2.7 Portable Trust, Certification, and Federation Maturity

- [ ] **TRUST-01**: Enterprise identity context propagates into portable
  credentials and federated trust artifacts without silently widening trust.
- [ ] **TRUST-02**: Agent Passport lifecycle includes status, revocation,
  supersession, and distribution semantics that relying parties can consume.
- [ ] **TRUST-03**: Certification registry becomes a multi-operator discovery
  surface, not just a local artifact store.
- [ ] **TRUST-04**: Cross-org reputation distribution stays conservative,
  explicit, and evidence-backed rather than inventing unsupported global trust
  claims.
- [ ] **TRUST-05**: Portable-trust docs, verifier tooling, and federation
  surfaces align to one supported issuance and presentation contract.

## Future Planned Milestone: v2.8 Risk, Attestation, and Launch Closure

- [ ] **RISK-01**: Insurer-facing behavioral feed/export exists with a stable
  schema and operator documentation.
- [ ] **RISK-02**: Attested runtime evidence can tighten or loosen issuance,
  approval, or budget policy through an explicit assurance tier.
- [ ] **RISK-03**: Formal proof debt and protocol/spec drift are reduced to a
  consciously accepted minimum, including closure of the remaining Lean `sorry`
  or an explicit scoped deferral.
- [ ] **RISK-04**: GA checklist, release audit, and partner proof artifacts
  close to an actual launch decision instead of a conditional candidate.
- [ ] **RISK-05**: Standards and external launch materials align to ARC rather
  than the old PACT identity.

## Out of Scope

| Feature | Reason |
|---------|--------|
| ARC as a direct payment rail | ARC is the authorization/evidence layer, not the settlement network. |
| Public certification marketplace in the rename milestone | Depends on portable-trust, registry, and discovery maturity after v2.5. |
| Multi-region Byzantine consensus | Premature for the current trust-control deployment model. |
| Full OS sandbox manager | Different product layer from ARC protocol and trust-control work. |
