## Current Milestone: v2.3 Production and Standards

**Milestone Goal:** Turn the current broad feature surface into a
release-qualifiable, maintainable, standards-aligned product by cleaning
release inputs, hardening deployment and observability, aligning the protocol
spec to shipped behavior, and preparing launch/submission artifacts.

**Phase Numbering:**
- Integer phases 21-24: planned v2.3 milestone work
- Future milestones continue from phase 25 onward

- [x] **Phase 21: Release Hygiene and Codebase Structure** - Remove release-input
  debt, enforce source-only packaging, and reduce structural sprawl in the
  largest runtime surfaces. (completed 2026-03-25)
- [x] **Phase 22: Qualification, Deployment, and Upgrade Hardening** - Make the
  release gates, deployment flows, and upgrade/rollback procedures concrete and
  repeatable. (completed 2026-03-25)
- [x] **Phase 23: Observability and Protocol v2 Alignment** - Add production
  diagnostics and align the protocol specification to the behavior the code
  actually ships. (completed 2026-03-25)
- [x] **Phase 24: Standards Submission and Launch Readiness** - Align SDK/docs
  to the production contract and produce the standards and launch artifacts
  needed for a credible release candidate. (completed 2026-03-25)

## Phase Details

### Phase 21: Release Hygiene and Codebase Structure
**Goal**: Release inputs come from source only, and the largest runtime surfaces
are modular enough to be maintained safely.
**Depends on**: Phase 20 and archived v2.2
**Requirements**: PROD-07, PROD-08
**Success Criteria** (what must be TRUE):
  1. Generated/build/cache artifacts are removed from tracked release inputs and
     package guards prevent them from coming back.
  2. Packaging manifests and ignore rules produce reproducible release inputs
     from source only.
  3. Oversized runtime entrypoints are split into clearer module boundaries
     without changing behavior.
  4. Regression coverage proves the hygiene and structure work did not widen
     trust or break shipped behavior.
**Plans**: 3 plans

Plans:
- [x] 21-01: Inventory tracked release-input debt and define cleanup/refactor
  boundaries.
- [x] 21-02: Remove generated artifacts and add package/CI hygiene guards.
- [x] 21-03: Refactor oversized runtime surfaces into maintainable modules with
  regression coverage.

### Phase 22: Qualification, Deployment, and Upgrade Hardening
**Goal**: Release qualification and operator deployment are scripted, documented,
and repeatable instead of tribal knowledge.
**Depends on**: Phase 21
**Requirements**: PROD-09, PROD-10
**Success Criteria** (what must be TRUE):
  1. A documented release-qualification lane passes across workspace, dashboard,
     SDK, and conformance-critical checks.
  2. Release scripts and CI expose failures clearly enough to be actionable
     without local guesswork.
  3. Trust-control and related operator surfaces have documented deployment,
     configuration, backup/restore, and upgrade/rollback procedures.
  4. Qualification and deployment documentation are specific enough to run a
     release candidate from scratch.
**Plans**: 3 plans

Plans:
- [x] 22-01: Define the release qualification matrix and acceptance gates.
- [x] 22-02: Implement automation/scripts/CI checks and fix blockers until the
  gates pass.
- [x] 22-03: Write deployment, configuration, backup/restore, and
  upgrade/rollback runbooks.

### Phase 23: Observability and Protocol v2 Alignment
**Goal**: Operators can diagnose failures in production, and the protocol spec
accurately matches the product that exists.
**Depends on**: Phase 22
**Requirements**: PROD-11, PROD-12
**Success Criteria** (what must be TRUE):
  1. Trust-control, remote runtime, A2A, and federation-critical paths expose
     health and diagnostic surfaces that are useful in production.
  2. Operators can triage common failure classes without reading source or
     replaying raw traffic by hand.
  3. `spec/PROTOCOL.md` v2 accurately describes shipped receipts, portable
     trust, federation, A2A, and certification behavior.
  4. Compatibility assumptions and any intentional spec/runtime gaps are
     documented explicitly rather than implied.
**Plans**: 3 plans

Plans:
- [x] 23-01: Define the observability and health contract for critical runtime
  surfaces.
- [x] 23-02: Implement and document production diagnostics plus regression
  coverage.
- [x] 23-03: Align `spec/PROTOCOL.md` v2 to shipped behavior and compatibility
  expectations.

### Phase 24: Standards Submission and Launch Readiness
**Goal**: SDKs, docs, standards artifacts, and launch evidence align to a single
production contract.
**Depends on**: Phase 23
**Requirements**: PROD-13, PROD-14
**Success Criteria** (what must be TRUE):
  1. SDKs, examples, and operator docs align with the spec v2 production
     contract.
  2. Standards-submission artifacts exist for receipts and portable trust, with
     clear terminology, scope, and non-goals.
  3. A GA launch checklist and risk register capture what is done versus what
     is still deferred.
  4. The milestone exits with release-candidate evidence, not just roadmap
     claims.
**Plans**: 3 plans

Plans:
- [x] 24-01: Align SDKs, examples, and operator docs to the v2 production
  contract.
- [x] 24-02: Prepare standards-submission artifacts and terminology mapping for
  receipts and portable trust.
- [x] 24-03: Assemble the GA checklist, risk register, and milestone closeout
  evidence.
