# Plan 379-02

## Objective

Close `OPER-02` by making the live `arc-tower` request path bind raw request
bytes into `EvaluationInput` on supported runtime bodies, then proving that the
same bytes are replayed downstream after evaluation.

## Tasks

1. Treat `crates/arc-tower/src/service.rs` as the execution seam and keep the
   existing body collection helper, but stop relying on `ReqBody: From<Bytes>`
   as an unexamined generic promise. Replace it with an explicit replay
   strategy for supported body types, or narrow the service API so supported
   body types are stated truthfully.
2. Thread the buffered raw bytes through one authoritative path:
   collect the request body once, compute SHA-256 and byte length from those
   exact bytes, pass the resulting `body_hash` / `body_length` into
   `EvaluationInput`, and rebuild the request body from the same bytes before
   forwarding to the inner service.
3. Keep the evaluator contract unchanged except for proving the propagation:
   add focused assertions in `crates/arc-tower/src/evaluator.rs` and/or
   `crates/arc-tower/src/service.rs` that non-empty request bodies change the
   signed receipt content binding, while truly empty bodies remain `None` / `0`.
4. Upgrade the Axum integration coverage in
   `crates/arc-tower/tests/axum_integration.rs` so the live middleware path
   handles a real `axum::body::Body` payload, the handler receives the original
   bytes after ARC evaluation, and allow/deny behavior still works with replayed
   bodies.
5. Make the Tonic coverage truthful in
   `crates/arc-tower/tests/tonic_integration.rs`: either run the middleware with
   an actual `tonic::body::Body` request type or rename the existing test to
   indicate it is only a generic Tower/HTTP2 approximation. Do not claim
   `OPER-02` complete with `Full<Bytes>`-only gRPC coverage if the production
   Tonic body type still cannot be replayed by the middleware.

## Constraint

`ArcService<S>` currently requires `ReqBody: Body + From<Bytes> + Send +
'static`. That works for `axum::body::Body` and the current `Full<Bytes>` test
body, but it does not describe real Tonic body plumbing. If phase `379` is
meant to close `OPER-02` for actual Tower runtime paths, this slice must do one
of the following explicitly:

- add a narrow body replay abstraction or constructor hook so `ArcService` can
  rebuild supported non-`From<Bytes>` body types such as `tonic::body::Body`
- or narrow the public/runtime claim to the body types that really are
  replayable today and leave real Tonic support as unresolved follow-on work

The first option is the preferred closeout path because it makes the runtime
claim and the tests line up.

## Acceptance

- `OPER-02` is satisfied only when the runtime middleware binds real raw body
  bytes into evaluation, not helper-only unit inputs.
- The forwarded request body is byte-for-byte identical to the original payload
  after ARC evaluation.
- Receipt/input content binding differs for distinct non-empty payloads and does
  not report `None` / `0` except for genuinely empty bodies.
- Axum coverage proves the real runtime path.
- Tonic coverage is either real for the supported body type or explicitly
  documented as still open; no misleading completion claim remains.
