# v2.82 Roadmap Snapshot

## v2.82 Normative Protocol Specification and Conformance

**Status:** complete locally 2026-04-13
**Goal:** Produce a specification that a third party could use to implement a
compatible ARC kernel or agent client, with formal message schemas, security
analysis, and a conformance suite that covers native ARC protocol behavior.

## Phases

- Phase `311`: Wire Protocol Specification -- complete
- Phase `312`: Version Negotiation and Error Taxonomy -- complete
- Phase `313`: Threat Model and Security Specification -- complete
- Phase `314`: Native Protocol Conformance and Standards Track -- complete

## Key Outcomes

- `spec/WIRE_PROTOCOL.md` now defines normative ARC framing, message semantics,
  and sequence diagrams, backed by checked-in schemas under
  `spec/schemas/arc-wire/v1`.
- Version negotiation and numeric ARC error registry artifacts are published
  under `spec/versions/arc-protocol-negotiation.v1.json` and
  `spec/errors/arc-error-registry.v1.json`, with runtime checks tied back to
  the shipped implementation.
- `spec/SECURITY.md` and `spec/security/arc-threat-model.v1.json` capture the
  agent-kernel-tool threat model, mitigations, residual risk, and transport
  security requirements.
- `crates/arc-conformance` plus `tests/conformance/native/` provide a
  language-neutral native conformance lane, while
  `spec/ietf/draft-arc-protocol-00.md` and
  `docs/standards/ARC_PROTOCOL_ALIGNMENT_MATRIX.md` package the milestone for
  external implementers and standards consumers.
