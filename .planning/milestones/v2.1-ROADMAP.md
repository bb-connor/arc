## Phase Details

### Phase 13: Enterprise Federation Administration
**Goal**: Enterprise identity providers can be onboarded and governed as first-class portable-trust sources instead of ad hoc bearer/JWT issuers.
**Depends on**: Phase 12 and shipped portable-trust alpha
**Requirements**: FED-01, FED-02
**Success Criteria** (what must be TRUE):
  1. An operator can configure a provider-admin surface for enterprise federation inputs, not just raw bearer verification settings.
  2. SCIM and SAML-backed identity context can be normalized into stable ARC principals and subject keys without widening trust silently.
  3. Trust-control policy can allow or deny portable-trust admission using provider, tenant, organization, role, and group context.
  4. Admin and operator surfaces show identity provenance clearly enough to debug cross-org authorization decisions.
**Plans**: 4 plans

Plans:
- [x] 13-01: Define the enterprise federation admin model and add provider configuration types for SCIM, SAML, and provider-managed identity sources.
- [x] 13-02: Implement normalized principal and subject derivation for SCIM/SAML-backed identity context with fail-closed parsing and trust rules.
- [x] 13-03: Extend trust-control policy and admission paths to gate on provider, tenant, organization, role, and group attributes.
- [x] 13-04: Add admin/reporting surfaces, docs, and integration tests for enterprise federation onboarding and debugging.

### Phase 14: Portable Verifier Distribution and Replay Safety
**Goal**: Portable verifier policy and challenge handling become reusable, signed, and replay-safe across processes and deployments.
**Depends on**: Phase 13
**Requirements**: VER-01, VER-02, VER-03
**Success Criteria** (what must be TRUE):
  1. Verifier policy can be created, signed, distributed, and referenced as a reusable artifact rather than only embedded inline.
  2. Challenge replay state is persisted and enforced across process restarts and multi-request verifier flows.
  3. Challenge create/respond/verify semantics are explicit across local CLI and remote verifier surfaces.
  4. Verification results explain both structural validity and policy acceptance cleanly enough for operators and relying parties.
**Plans**: 4 plans

Plans:
- [x] 14-01: Define signed verifier-policy artifact format, storage, and reference semantics.
- [x] 14-02: Implement persisted verifier challenge state with replay-resistant consumption and expiry enforcement.
- [x] 14-03: Extend CLI and trust-control verifier APIs to use stored verifier policies and replay-safe challenge flows.
- [x] 14-04: Add verification reporting, docs, and end-to-end tests for reusable verifier policy plus replay protection.

### Phase 15: Multi-Issuer Passport Composition
**Goal**: ARC defines truthful semantics for portable credential bundles issued by more than one authority without weakening current single-issuer guarantees.
**Depends on**: Phase 14
**Requirements**: PASS-01, PASS-02
**Success Criteria** (what must be TRUE):
  1. Multi-issuer passport composition semantics are explicitly defined and enforced in code instead of implicitly rejected or loosely accepted.
  2. Verifier evaluation explains which issuer and credential caused acceptance or rejection.
  3. Composition rules preserve existing single-credential verification guarantees and never invent aggregate truth not supported by the underlying credentials.
**Plans**: 3 plans

Plans:
- [x] 15-01: Freeze the multi-issuer passport composition model and acceptance rules in `arc-credentials`.
- [x] 15-02: Implement issuer-aware evaluation and reporting for composed passport bundles.
- [x] 15-03: Add CLI guides, fixtures, and regression tests covering accepted, rejected, and mixed multi-issuer bundles.

### Phase 16: Cross-Org Shared Evidence Analytics
**Goal**: Operators can analyze federated delegation and shared evidence references across org boundaries without copying every remote package into local receipt storage first.
**Depends on**: Phase 15
**Requirements**: FED-03, XORG-01, XORG-02
**Success Criteria** (what must be TRUE):
  1. Trust-control and dashboard can reference shared remote evidence packages directly where full import is unnecessary.
  2. Cross-org operator reports and comparison/drift views attribute activity across local and remote delegation chains.
  3. Federated issuance preserves upstream provenance through downstream operator analytics and reporting.
  4. Shared-reference flows stay fail-closed when remote evidence, proof coverage, or federation policy constraints are missing.
**Plans**: 4 plans

Plans:
- [x] 16-01: Define shared remote evidence reference model and trust-control APIs for reference-based federation surfaces.
- [x] 16-02: Extend receipt and operator analytics to consume local plus remote evidence references truthfully.
- [x] 16-03: Wire dashboard and CLI comparison/reporting surfaces to shared remote evidence references.
- [x] 16-04: Add end-to-end tests and docs for cross-org shared-evidence analytics and downstream provenance reporting.
