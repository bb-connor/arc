## Active Milestone: v2.5 ARC Rename and Identity Realignment

`v2.4` is archived. `v2.5` is now the active milestone, with `v2.6` as the
next planned wave.

**Milestone Goal:** Rename the project and product from PACT to ARC across
repo, crate, package, CLI, doc, protocol, and portable-trust surfaces while
preserving migration clarity, artifact verifiability, and operator continuity.

**Why now:** The post-v2.4 architecture is steady enough to support a wide
rename. `docs/research/DEEP_RESEARCH_1.md` also gives a stronger strategic
identity for the product under ARC than the older transport-first ARC
expansion. The rename needs to happen before the next feature wave so payment,
portable-trust, and insurer-facing work lands under one coherent name.

**Phase Numbering:**
- Integer phases 29-32: active `v2.5` milestone work
- Future milestones continue from phase 33 onward

- [x] **Phase 29: Rename Inventory and Compatibility Contract** - Audit every
  ARC-branded surface and decide whether it renames, aliases, or stays frozen
  with rationale.
- [x] **Phase 30: Workspace, SDK, CLI, and Repo Surface Rename** - Move the
  repo, crates, package metadata, SDK names, and CLI surface from `arc` to
  `arc` with explicit migration support.
- [x] **Phase 31: Protocol, Artifact, and Identity Migration** - Resolve
  schema, route, config, artifact-family, and `did:arc` migration semantics.
- [x] **Phase 32: Qualification, Documentation, and Launch Narrative** - Re-run
  the full qualification lane, update docs/specs/standards, and publish the ARC
  migration story.

## Phase Details

### Phase 29: Rename Inventory and Compatibility Contract

**Goal**: Turn the rename into an explicit compatibility program instead of an
unsafe string-replacement exercise.  
**Depends on**: Phase 28 and archived v2.4  
**Requirements**: ARC-04, ARC-05, ARC-06  
**Why first**: The repo already has signed artifacts, stable CLI workflows,
SDKs, standards drafts, and portable-trust identifiers. Those boundaries need a
deliberate migration matrix before any mass rename starts.

**Primary surfaces to classify:**
- repo and package metadata
- workspace members and crate names
- CLI binary, subcommands, examples, and release scripts
- SDK package names and import paths
- protocol schema IDs, headers, route prefixes, env vars, and config keys
- `did:arc`, passport, verifier-policy, certification, and evidence artifact
  families

**Success Criteria** (what must be TRUE):
1. Every `ARC` / `arc` surface is inventoried and classified as rename,
   alias, freeze, or convert.
2. The `did:arc` transition contract is decided before signed portable-trust
   artifacts are renamed.
3. A migration matrix exists for operators, SDK consumers, and downstream
   integrators.
4. The milestone can proceed without ambiguous half-renamed surfaces.

**Plans**: 3 plans

Plans:
- [ ] 29-01: Audit the repo and produce a rename/alias/freeze inventory.
- [ ] 29-02: Decide the artifact and identity migration contract, including
  `did:arc`.
- [ ] 29-03: Publish the migration matrix and operator/developer rollout order.

### Phase 30: Workspace, SDK, CLI, and Repo Surface Rename

**Goal**: Rename the visible developer/operator surfaces to ARC while keeping a
stable migration path.  
**Depends on**: Phase 29  
**Requirements**: ARC-01, ARC-02, ARC-03  
**Why second**: Once the compatibility rules are decided, the highest-value
rename work is the visible workspace and package surface.

**Planned moves:**
- `arc-*` workspace crates -> ARC-aligned crate and package names
- `packages/sdk/arc-ts`, `arc-py`, `arc-go` -> ARC-aligned package surfaces
- `@arc-protocol/sdk` -> ARC-aligned npm identity
- `arc` CLI binary and operator examples -> `arc`, with explicit alias or
  major-version migration behavior
- repo metadata, release artifact labels, scripts, and branding -> ARC

**Success Criteria** (what must be TRUE):
1. The repo, workspace, crates, SDK metadata, and CLI present ARC as the
   primary identity.
2. Supported legacy `pact` aliases behave exactly as documented.
3. Downstream migration instructions exist for Rust, TS, Python, and Go
   consumers.
4. Qualification still passes after the rename.

**Plans**: 3 plans

Plans:
- [ ] 30-01: Rename repo/workspace/package metadata and add compatibility shims.
- [ ] 30-02: Rename SDK and CLI surfaces and update release automation.
- [ ] 30-03: Requalify renamed workspace/package flows before protocol-level
  changes.

### Phase 31: Protocol, Artifact, and Identity Migration

**Goal**: Resolve the hard rename edges where external semantics matter:
protocol markers, signed artifact families, and portable-trust identity.  
**Depends on**: Phase 30  
**Requirements**: ARC-04, ARC-05, ARC-06  
**Why third**: These surfaces are the easiest place to accidentally break
verifiability, so they should move after the visible package/CLI rename is
stable.

**Primary surfaces:**
- spec and standards profile identifiers
- schema IDs and file markers
- route prefixes, headers, env vars, and config keys
- receipt, passport, verifier-policy, certification, and evidence package
  identifiers
- `did:arc` or its ARC successor/compatibility mode

**Success Criteria** (what must be TRUE):
1. There is no ambiguous mix of ARC and ARC protocol/artifact names left in
   supported surfaces.
2. Legacy ARC artifacts either verify/import cleanly or have documented
   conversion tooling.
3. Identity and portable-trust docs match the actual shipped migration model.
4. Spec and implementation align again after the rename.

**Plans**: 3 plans

Plans:
- [x] 31-01: Rename or freeze protocol markers and schema families explicitly.
- [x] 31-02: Implement artifact verification/import compatibility or converters.
- [x] 31-03: Close spec and portable-trust identity drift around the new ARC
  contract.

### Phase 32: Qualification, Documentation, and Launch Narrative

**Goal**: Prove the renamed ARC surface works end-to-end and make the docs tell
one coherent story again.  
**Depends on**: Phase 31  
**Requirements**: ARC-07, ARC-08  
**Why last**: The qualification lane and public/operator docs should validate
the final renamed surface, not a mid-transition snapshot.

**Primary outputs:**
- updated README, VISION, Strategic Roadmap, release docs, and standards drafts
- migration guide for operators and SDK consumers
- qualification and conformance evidence covering ARC plus any supported ARC
  aliases
- launch/positioning language aligned to "attested rights, bounded spend, and
  verifiable receipts"

**Success Criteria** (what must be TRUE):
1. Docs, standards, release, and operator guidance all describe ARC instead of
   a mixed ARC/ARC product.
2. The release lane proves the renamed ARC surface end-to-end.
3. Migration docs make the rename survivable for current users.
4. The milestone closes with one coherent product narrative and one coherent
   technical contract.

**Plans**: 3 plans

Plans:
- [x] 32-01: Rewrite docs/spec/release materials to the ARC identity.
- [x] 32-02: Run full qualification and conformance on the renamed surface.
- [x] 32-03: Assemble the migration package and milestone closeout evidence.
